# Benchmark Agent Harness Implementation Plan (Plan 2)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Build the complete agent harness — the `run_bench.py` driver (Phase 2) and the `grade.py` grading pipeline (Phase 3) — and validate it end-to-end on a 16-cell graded smoke grid on bank-chat.

**Architecture:** A condition-agnostic driver plus a decoupled grader. `bench/run_bench.py` expands the grid and dispatches each cell through `bench/claude_runner.py`, which assembles the `claude -p` argv from Plan 1's loaders, spawns it with the four de-risk corrections (driver-side turn cap, `--verbose` + `stdin=DEVNULL`, subprocess `cwd`=checkout, enforcement monitored via `tool_call_breakdown`), parses stream-json, and returns a `CellResult` (`grade=null`). `bench/grade.py` then consumes `cells.jsonl` + transcripts + the oracle, dispatches by each question's `grading` field to programmatic graders or a condition-blinded glm-5.2 LLM judge, computes Cohen's κ over the judged subset, and writes `graded.jsonl` with `grade` filled.

**Tech Stack:** Python 3 (`.venv/bin/python`), pytest, PyYAML, the `claude` CLI headless (`-p`), the `java-codebase-rag` operator CLI for index builds (already done in Plan 1).

## Global Constraints

- Use `.venv/bin/python` and `.venv/bin/pip` only — never system `python`/`pip`. Editable install of `java-codebase-rag` is enforced by `tests/conftest.py`.
- `bench/` is a standalone package imported as `import bench.<module>`; never fold it into `java_codebase_rag`. `tests/bench/` deliberately has NO `__init__.py` (re-adding it shadows the source package).
- Run only the relevant pytest subset during development; run the full `tests/bench/` suite once at the end.
- Bench indexes/checkouts live under `bench/indexes/` and `bench/checkouts/` (gitignored, built in Plan 1). Plan 2 adds `bench/results/` (also gitignored).
- Always pass jrag paths as **absolute**. The MCP `command` must be `sys.executable`, not bare `python`.
- `claude -p` requires `--verbose` with `--output-format stream-json`, and its stdin must be closed (`stdin=DEVNULL`). There is no `--max-turns` flag — the turn cap is enforced driver-side. There are no temperature/seed flags — `seed`/`temperature` are recorded as intended metadata only.
- The LLM judge runs as a single-turn `claude -p --model glm-5.2 --output-format json` call (no tools, no MCP, no cap). It is independent of the subject models.
- Every task commits with `Co-Authored-By: Claude <noreply@anthropic.com>` at the end of the message.

```
bench/
  claude_runner.py              # CellSpec -> argv, spawn claude -p, turn cap, parse stream -> CellResult
  run_bench.py                  # grid expansion + dispatch + results I/O (idempotent, resumable) + CLI
  grade.py                      # programmatic graders + blinded glm-5.2 judge + cohen_kappa + CLI

  test_claude_runner.py         # stream parse, mcp materialize, argv assembly, run_cell + cap
  test_run_bench.py             # grid expansion, results I/O, idempotency/resume, CLI
```

**Responsibility split.** `claude_runner.py` owns one cell end-to-end (argv, spawn, cap, parse, `CellResult`). `run_bench.py` owns the grid (expansion, dispatch, results I/O, resume, CLI) and depends only on `claude_runner`'s public surface. `grade.py` owns scoring (graders, judge, κ, CLI) and depends only on the cell JSONL + transcript + oracle — it never imports `claude_runner` or `run_bench`.

**Interfaces:**
- Consumes: nothing (first task).
- Produces:
  - `@dataclass(frozen=True) class StreamSummary` with fields: `tool_call_breakdown: dict[str,int]` (tool_use name → count), `context_bytes_retrieved: int` (sum of tool_result content lengths), `n_turns: int` (count of `assistant` stream events), `tokens: dict` (`{"input": int, "output": int, "total": int}`; `total` = input+output), `stop_reason: str | None`, `terminal_reason: str | None`, `is_error: bool`, `api_error_status: str | None`, `final_answer: str | None` (the `result.result` field), `num_turns_reported: int | None` (the `result.num_turns` field).
  - `parse_stream(lines: Iterable[str]) -> StreamSummary` — single-pass over stream-json lines. Behavior:
    - Skip non-JSON / blank lines silently.
    - For an event with `type == "assistant"`: increment `n_turns`; for each `message.content[]` entry of `type == "tool_use"`, increment `tool_call_breakdown[name]`.
    - For an event with `type == "user"`: for each `message.content[]` entry of `type == "tool_result"`, add the length (in characters) of its `content` (stringified) to `context_bytes_retrieved`.
    - For an event with `type == "result"`: populate `stop_reason`, `terminal_reason`, `is_error`, `api_error_status`, `final_answer` (= `result`), `num_turns_reported` (= `num_turns`), and `tokens` from `usage` (`input_tokens`, `output_tokens`; `total` computed). If no `result` event is present, those fields stay `None`/defaults (the capped/truncated case).
    - Events with `type == "system"` are ignored.
